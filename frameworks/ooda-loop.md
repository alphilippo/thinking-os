# OODA Loop

## Quand l'utiliser

- Contexte compétitif rapide (concurrence, négociation, crise)
- Tu dois décider vite avec info incomplète
- Un événement extérieur vient de changer la donne
- L'utilisateur mentionne un concurrent, une attaque, un imprévu

## Procédure

Le cycle OODA — Observe, Orient, Decide, Act — doit tourner plus vite que celui de l'adversaire (ou de la situation).

1. **Observe** : quelles sont les données brutes nouvelles ? (faits seulement, pas d'interprétation)
2. **Orient** : comment ces données modifient ma carte mentale ? Quels biais/narratifs pourraient me tromper ici ?
3. **Decide** : parmi les options actionnables maintenant, laquelle a la meilleure asymétrie risque/récompense ?
4. **Act** : exécuter la décision — même imparfaite, car l'inaction = se laisser dépasser par la boucle de l'adversaire.
5. **Recommencer** : tout signal de retour (feedback, réaction adverse) déclenche un nouveau cycle.

## Format synthèse

```
🧠 Framework : OODA Loop
📌 Pourquoi ce cadre : situation rapide, il faut agir avant de sur-analyser

Observe (faits nouveaux) :
- <fait 1>
- <fait 2>

Orient (re-lecture) : <en 1-2 phrases, la nouvelle carte>
Biais à surveiller : <un biais spécifique>

Decide : <option retenue parmi les alternatives>

✅ Action : <action à faire dans les prochaines 24-72h>
⚠️ Angle mort : <signal à surveiller pour relancer la boucle>
```

## Format deep-dive

1. **État initial** : carte mentale avant l'événement déclencheur
2. **Observe** : liste complète des faits nouveaux, avec source et niveau de confiance
3. **Orient** :
   - Narratif possible #1 (+ évidence)
   - Narratif possible #2 (+ évidence)
   - Narratif le plus probable actuellement
   - Biais de confirmation à surveiller
4. **Decide** : 3-4 options d'action + asymétrie de chacune
5. **Act** : action retenue, timeline, owner, ressources
6. **Prochain cycle** : quels signaux re-lancent la boucle ? à quelle fréquence observer ?

## Exemple bref

**Situation** : *"Un concurrent direct vient d'annoncer une feature identique à la nôtre, 2 semaines avant notre lancement"*

Observe :
- Annonce faite hier sur LinkedIn, 300 réactions
- Ils n'ont pas encore livré (juste "coming soon")
- Leur base client est 3x la nôtre

Orient : le marché va associer cette feature à eux, pas à nous, sauf si on change de timing ou de narratif. Biais à surveiller : sunk cost sur notre plan de lancement.

Options :
1. Accélérer et lancer avant eux → risque qualité
2. Changer le narratif (pas "nouvelle feature" mais "réponse au besoin X") → repositionnement
3. Attendre qu'ils livrent et attaquer sur la qualité d'exécution → perte d'initiative

Decide : option 2. On pivote le messaging autour d'un angle qu'ils n'ont pas.

✅ Action : réécrire la landing page + email de lancement dans les 48h.
⚠️ Angle mort : leur "coming soon" peut être du vapor-ware, surveiller leur blog.

## Pièges à éviter

- Ne pas bloquer sur "Observe" par peur de décider. La boucle tourne, même avec info imparfaite.
- Ne pas agir par réflexe sans "Orient". OODA n'est pas "fonce tête baissée".
- Le framework n'est pas un mono-shot — il doit boucler. Si tu l'utilises une fois et tu ranges, tu l'as mal utilisé.
- Dans un contexte non-compétitif (décision personnelle profonde, apprentissage), ce n'est PAS le bon framework.
